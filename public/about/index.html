      About

      Gitflow Hands-on Playground

      チーム全員が Gitflow を体験しながら、Hugo + Hextra のサイト編集を学ぶための実践用リポジトリです。
      作業は dev ブランチ を基点に、各自の feature/<username> ブランチで行い、PR で dev に統合します。レビュー後、管理者が main に反映します。

      0. 前提

      • Git 使用法と GitHub アカウントを登録済み

      • （推奨）SSH 接続設定済み（または HTTPS で OK）

      • ローカルで Hugo が動くこと（hugo version が通ればOK / Extended推奨）

      1) リポジトリの取得と初期設定 

      # 1. クローン（SSH の例） 
git clone git@github.com:<OWNER>/<REPO>.git 
cd <REPO>  
# 2. 最新の dev を取得 
git fetch origin 
git checkout dev 
git pull origin dev  
# 3. 自分用の作業ブランチを作成（例: feature/pomworld） 
git switch -c feature/<username>

      以降の作業は 必ず自分の feature/<username> ブランチ で行ってください。

      2) ローカルプレビュー（Hugo）

      # ターミナルでローカルサーバーを起動 hugo server -D

      • ブラウザで http://localhost:1313/ を開くとプレビューできます。

      • このリポジトリは Hugo の Hextra テーマのクイックスタート構成を使用しています。 IMFingGitHub

      3) 課題A：トップページに自分のカードを追加

      対象ファイル: content/_index.md

      1. 既存のカードセクション（

        ...
        ）があれば、その中に自分のカードを1枚追加。無ければ下記のように新規ブロックを作ってOK。

      2. タイトルは自分の GitHub ユーザー名、サブタイトルは自由、画像はURLでもローカルでも可。

      3. 自分のページ（後述の /members/<username>/）にリンクしてください。

      md

      コピーする

      <username>
      Hello Hextra!

      • image はリモートURLのほか、static/ 配下や assets/ 配下の画像も利用可能（method/options で画像処理も可）。 IMFing

      • cols で列数を調整できます（小さな画面では自動的に折り返されます）。 IMFing

      参考：Hextra の Cards ショートコード（使用例・パラメータ） IMFing

      4) 課題B：自分のメンバーページを作る

      新規ディレクトリとファイルを作成：

      bash

      コピーする

      content/members/<username>/_index.md

      最小サンプル（必要に応じて加筆）：

      md

      コピーする

      --- title: "<username>" type: docs sidebar: open: true --- # <username> 自己紹介やスキル、担当領域などを自由に書いてください。 - Twitter: … - Telegram: … - Link: …

      テーマのドキュメント系レイアウトを使いたい場合は type: docs を付ける運用が簡単です（サイト設定により不要な場合もあります）。テーマ設定やショートコード一覧は Hextra ドキュメントを参照してください。 IMFing+1IMFing+1

      5) 変更のコミットとプッシュ

      bash

      コピーする

      # 変更確認 git status git add . # 意味のあるコミットメッセージで git commit -m "feat(members): add <username> card and profile page" # 初回は upstream 設定込みでプッシュ git push -u origin feature/<username>

      6) PR を作成（feature → dev）

      • GitHub 上で base: dev / compare: feature/<username> を選んで PR 作成

      • PR の説明欄に：

        • 変更点の概要

        • スクショ or ローカルプレビュー確認事項

        • 自分のページへのリンク

      • PRを作成したら Telegram で管理者に連絡

      7) レビュー → 修正 → マージの流れ

      • レビューコメントが来たら、同じブランチに修正コミットしてプッシュすれば PR に自動で反映されます。

      • dev の更新を取り込みたい場合：

      bash

      コピーする

      git fetch origin git checkout feature/<username> # どちらでもOK（チーム標準に合わせる） git merge origin/dev # または git rebase origin/dev git push

      • 問題なければ管理者が dev にマージし、適切なタイミングで main に反映します（基本は管理者作業）。

      8) 完了後の片付け

      • マージ後は GitHub 上の 「Delete branch」 ボタンで feature/<username> を削除

      • ローカルでも削除してスッキリ：

      bash

      コピーする

      git branch -d feature/<username> git fetch -p

      トラブルシューティング（よくある）

      • Permission denied (publickey)
        → GitHub に SSH 公開鍵を登録するか、HTTPS の URL でクローン/プッシュしてください。

      • dev が進んでいてコンフリクト
        → 上の「merge / rebase」手順で origin/dev を取り込み、コンフリクト解消後に再コミット & プッシュ。

      • Hugo が動かない / 画像が出ない
        → hugo version で Extended になっているか確認。画像パスは static/ or assets/ の想定に合っているか見直し。 IMFing

      参加ルール（軽量）

      • ブランチ名：feature/<username> を厳守

      • コミットメッセージ：目的が伝わること（feat:, fix:, docs: 等は推奨）

      • PR の向き先は 必ず dev（main 直は不可）

      • PR 作成後は Telegram で声かけ

      • レビュワーが「OK」なら dev → main の反映は管理者が実施
